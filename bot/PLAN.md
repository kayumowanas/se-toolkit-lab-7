The bot will be built in layers so each task adds capability without forcing a rewrite. The first layer is a transport-independent handler layer: simple async functions that accept parsed input and return plain text. This is the core requirement for testability, because the same code must work both from Telegram updates and from the CLI `--test` mode. The second layer is the transport layer in `bot.py`, which translates either Telegram messages or a test string into handler calls. The third layer is the services layer, where backend and LLM integrations live behind small client classes. That keeps HTTP details and tool-calling logic out of handlers.

Task 1 creates the scaffold, configuration loading, placeholder handlers, and a router for slash commands and plain text. Task 2 replaces placeholder responses with real LMS backend calls and friendly error handling. Task 3 adds LLM-based intent routing with tool schemas for all required backend endpoints and feeds tool results back into the model. Task 4 containerizes the bot, adds it to Docker Compose, and documents deployment. This structure keeps logic modular, testable, and ready for the VM deployment workflow used by the lab.
